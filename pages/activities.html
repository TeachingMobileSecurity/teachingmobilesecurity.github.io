Activities

The Teaching Mobile Security project is comprised of separate course modules, which are designed to meet a wide range of experience levels ranging from new programmers, to seasoned mobile developers. In the following section, we will briefly detail each activity. A more thorough analysis may be found in the primary project repository.

Beginner

These activities are designed for students just learning about mobile development or security.

  • Protecting Private Info: You share a lot of data with the software that you use every day. However, do you ever think about this information that you are sharing? In this basic activity, particpants will be shown how easy it is for "sensative" information to be shared with tothers. [Repo]
    • Activities Access: Security issues arise when people try to access specific unauthorized activities. An example could be a bank app where users try to access a balance management activity without properly logging into the system. [Repo]
    • Intent Protection: Android uses "Intents" to pass data between apps, for examples between the Facebook and Facebook Messenger apps. Data passing between these apps may be easily (and improperly) read by other apps. This module explains how to protect information being sent via Intents between apps. [Repo]
    • XML: XML is very easy to read using reverse engineering. As a result, it is best to avoid saving important information, like Ads code or Map Code within XML files. [Repo]

    Medium

    The following activities are designed for students and developers who have a fair amount of experience with mobile development.

    • Secure Webviews: This demonstrates the negative implications of using JavaScript in Webview to pass data from an Android app to a server. This is considered bad practice because anyone could use malicious JavaScript code on their website to gain private user information associated with the app. [Repo]
    • Broadcast: Broadcast data sent by the app is easy to access from any other app in the system. Because of this, the data broadcasted to specific apps should be encrypted. Intercepted unencrypted Broadcasts could lead to serious security and privacy issues. [Repo]
    • Data Storage: When an app does not secure stored data, like data files, shared references, and databases (i.e. SQLite), it has the potential to be read by any other app. This means that important information stored in these files (such as database connection information) should be encrypted. [Repo]
    • Data Over HTTP: Data that moves over an unencrypted HTTP (Internet) connection is vulnerable to “Man in the Middle” attacks. One example of this is credit card information, which could be intercepted midstream, if sent over an unsecured connection. [Repo]
    • DoS: Denial of Service (DoS) attacks are a common problem with Android, because a malicious party could create an overwhelming number of HTTP requests directed at a specific server. The environment must be managed to make them less vulnerable to these types of attacks. [Repo]
    • Services: Services, particularly bound services, can be accessed by any app on a device if not properly protected. There must always be one or more ways of keeping a service secure and accessible only to apps we want trust. [Repo]

    Advanced

    These activities are intended to challenge even the most seasoned Android developers.

    • Ad Libraries: In-app advertisements (“Ads”) libraries are able to use all of the permissions given to the app containing the Ads library. This is true as well for those who did not give these permissions to the Ads library itself. This can lead to various security and privacy issues within the app, including Ad libraries collecting sensitive user information such as locations or contact info. [Repo]
    • Content Providers: Content providers share data between apps, and any app in the system can access the "Content_Providers" database. Because of this, data stored here must be kept secure and encrypted, so that it can only be read by an authorized app. [Repo]
