Workshop Summary

Want to learn about mobile Security? This workshop will introduce you to our mobile security modules which are oriented towards developers of all experience levels. We will be conducting several short activities related to mobile security. The activities will be:

Activities

  1. Protecting Private Info: You share a lot of data with the software that you use every day. However, do you ever think about this information that you are sharing? In this basic activity, particpants will be shown how easy it is for "sensative" information to be shared with tothers. [Repo]
  2. Activities Access: Security issues arise when people try to access specific unauthorized activities. An example could be a bank app where users try to access a balance management activity without properly logging into the system. [Repo]
  3. XML: XML is very easy to read using reverse engineering. As a result, it is best to avoid saving important information, like Ads code or Map Code within XML files. [Repo]
  4. Proper Permissions Usage: As a user and a developer, you need to be careful about the permissions your apps use. This activity will provide an understanding of proper permissions usage from both the perspective of the end user and of the developer. [Repo]
  5. Secure Webviews: This demonstrates the negative implications of using JavaScript in Webview to pass data from an Android app to a server. This is considered bad practice because anyone could use malicious JavaScript code on their website to gain private user information associated with the app. [Repo]
  6. Data Storage: When an app does not secure stored data, like data files, shared references, and databases (i.e. SQLite), it has the potential to be read by any other app. This means that important information stored in these files (such as database connection information) should be encrypted. [Repo]
  7. Data Over HTTP: Data that moves over an unencrypted HTTP (Internet) connection is vulnerable to “Man in the Middle” attacks. One example of this is credit card information, which could be intercepted midstream, if sent over an unsecured connection. [Repo]
  8. Ad Libraries: In-app advertisements (“Ads”) libraries are able to use all of the permissions given to the app containing the Ads library. This is true as well for those who did not give these permissions to the Ads library itself. This can lead to various security and privacy issues within the app, including Ad libraries collecting sensitive user information such as locations or contact info. [Repo]
  9. Content Providers: Content providers share data between apps, and any app in the system can access the "Content_Providers" database. Because of this, data stored here must be kept secure and encrypted, so that it can only be read by an authorized app. [Repo]
  10. App Reverse Engineering: Android apps are fairly easy to reverse engineer and modify. In this example, we will reverse engineer a basic Android application and demonstrate exploitations that could be performed on the app.
